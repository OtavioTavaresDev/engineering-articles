Designing a Scalable Vulnerability Scanning Pipeline

An asynchronous architecture case study on scanning thousands of repositories with isolated environments and professional reporting.

April 28, 2026 16 min read

We needed to scan thousands of repositories — Python and Node.js — for known vulnerabilities, run the scans in isolated environments, and deliver PDF and Excel reports to security teams. The naive approach of performing the scan inside the request-response cycle collapsed the moment we hit a 10‑minute scan time on a mid‑size monorepo. What follows is a dissection of the architecture we settled on, the trade‑offs we made, and the scars we earned along the way.

The Sync Trap: Why Not async def

FastAPI is built on ASGI and can handle thousands of concurrent connections via cooperative multitasking. That works beautifully for I/O-bound workloads. But a vulnerability scan is CPU‑ and I/O‑intensive in an unbounded way:

  • • Cloning or updating a repo can saturate disk I/O for minutes.
  • • Running pip-audit or npm audit inside Docker consumes CPU unpredictably.
  • • Generating richly formatted PDFs from a large dependency tree is itself a heavy task.

⚡ Quick Math

A single scan taking 8 minutes holds the connection open, exhausts gunicorn/uvicorn worker slots, and pushes p99 latency into the thousands of seconds. HTTP timeouts will fire long before the scan finishes.

Architecture at a Glance

[Client] │ ▼ [FastAPI] ──enqueue──▶ [Redis (RQ)] │ │ ◄──202 + scan_id ▼ [Worker (RQ)] │ ├─ docker run ├─ parse results (JSON) ├─ build PDF / Excel └─ store in S3, update DB ▲ ◄── poll / webhook ──────────┘

🔧 FastAPI

Async endpoints for lightweight polling and webhooks without blocking worker threads.

📦 Redis + RQ

Minimal API, function-centric scan jobs. Chosen over Celery for simplicity.

🐳 Docker

Every scan in its own throwaway container. Malicious setup.py cannot touch the worker filesystem.

📄 ReportLab + OpenPyXL

PDF and Excel generation inside dedicated worker queue with higher memory limits.

Performance Bottlenecks & Mitigations

Docker Image Pull Latency

Every new worker node pulls an 800MB scanner image on its first scan. We pre‑pull images at node startup using an init container in Kubernetes, and use a local registry mirror.

Queue Length Explosion

A sudden submission of 500 repos can swamp the scan queue. We use rate limiting at the API gateway, two priority queues (scans-high for users, scans-bulk for CI), and a hard cap on queue depth returning HTTP 503 with Retry-After.

Worker Saturation

Docker containers on the same host compete for CPU/IO. We cap concurrent Docker containers per host using a BoundedSemaphore inside the worker, coupled with Kubernetes pod anti‑affinity and HPA scaling based on queue length.

Failure Scenarios & Recovery

💥 Worker Hard‑Crash Mid‑Scan

Stuck‑job reaper runs every 2 minutes, identifies jobs stuck over 40 minutes, removes them from the started registry, and re‑enqueues them.

⏱️ Scan Timeout

container.wait(timeout=1800) kills the container gracefully. We deliberately do not use RQ's job_timeout — it sends SIGKILL to the entire worker.

💾 Redis Out‑of‑Memory

Max job payload size enforced at API layer. volatile‑lru eviction policy. Results never stored in Redis — only lightweight S3 path in database.

What We Would Do Differently

  • Use a task system with native priority and dead‑letter‑queues (Celery or SQS‑based pipeline) instead of building our own reaper.
  • Stream results back rather than storing giant JSON blobs — push findings into an event stream so the report generator can start before the scan finishes.
  • Invest in a proper resource scheduler (Nomad or Kubernetes Jobs) that natively understands container lifecycles and disk quotas.
